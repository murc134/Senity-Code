1. Was ist die Senity CLI?

Die Senity CLI stellt einen globalen Befehl senity bereit, der den Senity-Workspace-Container ad-hoc startet und das aktuelle Arbeitsverzeichnis als /workspace/cwd in den Container mountet. Vorbild sind die CLIs claude, codex und gemini.

Use-Case: irgendwo im Dateisystem cd projekt-x, dann senity tippen, in der Senity-Claude-Session landen, am Code arbeiten, beim Exit verschwindet der Container, der Ordner bleibt unveraendert.

Wann nutzen? Spontane Sessions in fremden Projekt-Ordnern, schnelle Demos, Onboarding neuer Teammitglieder. Fuer persistente Multi-Repo-Workspaces mit eigener .bindings-Konfiguration bleibt der klassische claude-senity.sh/.bat-Launcher das richtige Werkzeug.

2. Installation

Linux / macOS

curl -fsSL https://git.senity.ai/senity-admin/senity-claude-code/raw/branch/main/senity-cli/install.sh | bash

Installiert nach ~/.local/bin/senity. Liegt der Pfad nicht im PATH, gibt der Installer die noetige Zeile fuer ~/.bashrc oder ~/.zshrc aus.

Windows (PowerShell 7+)

irm https://git.senity.ai/senity-admin/senity-claude-code/raw/branch/main/senity-cli/install.ps1 | iex

Installiert nach %USERPROFILE%\.senity\bin\senity.ps1 und legt einen senity.bat-Shim fuer cmd.exe an. Der User-PATH wird automatisch erweitert. Danach eine neue Shell oeffnen.

3. Erst-Login

senity login

Fragt nach Senity-Proxy-URL und Proxy-Key, schreibt nach ~/.senity/.env (Mode 600 auf Unix). Die URL hat den Default https://sdr.senity.ai/api/claude-proxy. Den Key liefert der Senity-Admin.

Sicherheit. Der Key landet im Klartext in ~/.senity/.env. Verzeichnis-Berechtigung ist 0700, Datei 0600. Niemals einchecken, niemals weitergeben.

4. Alltagsgebrauch

cd ~/projekte/foo
senity

Beim ersten Aufruf laeuft ein Auto-Update (Docker-Image pullen, Skills / Commands / Agents / MCPs klonen). Danach startet der Container, mountet ~/projekte/foo als /workspace/cwd und startet die Senity-Claude-Session. Beim Exit ist der Container weg.

Zusatz-Mounts

senity --mount ~/.gitconfig:/workspace/.gitconfig:ro
senity --mount /data/shared:/workspace/shared

Spezifisches Image-Tag

senity --image ghcr.io/murc134/senity-code:1.4.2

Workspace-HOME~/.senity/workspace → /workspacen/a

Update-Default-Philosophie. Auto-Update laeuft bei jedem Aufruf. Das spart das manuelle senity --update-Subcommand und garantiert, dass jede Session den aktuellen Skill- und Image-Stand sieht. Power-User koennen mit --skip-update bewusst offline arbeiten.

7. Konfiguration

Alle CLI-Daten liegen unter ~/.senity/: 

~/.senity/
+-- .env                 # SENITY_CHAT_PROXY_URL + KEY (chmod 600)
+-- mcp-config.json      # optional, User-globale MCP-Server
+-- workspace/           # persistentes /workspace (Claude-Settings, Codex-Tokens, ...)
+-- cache/
    +-- skills/          # Klon von murc134/Claude-Skills
    +-- commands/        # Klon von murc134/Claude-Commands
    +-- agents/          # Klon von murc134/Claude-Agents
    +-- senity-mcps/     # Klon von senity/senity-mcps

MCP-Server konfigurieren

Eigene MCP-Server oder Token-Overrides legt der User in ~/.senity/mcp-config.json ab. Format und Beispiel siehe mcp-config.example.json im Hauptrepo. Datei wird beim Start read-only in den Container gemountet und mit den Repo-Defaults gemergt.

8. Parallele Sessions

Jeder Aufruf bekommt einen eigenen Container-Namen senity-<pid>-<epoch>. Mehrere Terminals koennen ohne Konflikt parallel senity aufrufen, jeder bekommt seinen eigenen Container. Beim Exit verschwinden alle (--rm).

Der gemeinsame Mount ~/.senity/workspace ist read-write fuer beide Sessions. Vorsicht bei gleichzeitiger Aenderung derselben Settings-Datei, last-write-wins.

9. Update

Die Wrapper-Skripts updaten sich selbst nicht automatisch. Re-Run des Installers ueberschreibt das Script, Cache und .env bleiben: 

# Linux / macOS
curl -fsSL https://git.senity.ai/senity-admin/senity-claude-code/raw/branch/main/senity-cli/install.sh | bash

# Windows
irm https://git.senity.ai/senity-admin/senity-claude-code/raw/branch/main/senity-cli/install.ps1 | iex

10. Troubleshooting

"Image fehlt und --skip-update verhindert Pull"

Aufruf ohne --skip-update wiederholen, der Erstaufruf braucht einen Pull.

"docker pull fehlgeschlagen ... 401 Unauthorized"

Einmalig docker login git.senity.ai. Username = Gitea-User, Passwort = Personal Access Token mit Scope read:package (Gitea: User Settings -> Applications -> Generate New Token).

"SENITY_CHAT_PROXY_KEY fehlt"

senity login ausfuehren.

Skills oder Commands fehlen im Container

Auto-Update braucht SSH-Zugriff auf die Repos. SSH-Agent / Key in ~/.ssh/config pruefen. --skip-update nutzt den letzten erfolgreichen Cache-Stand.

Permission-Probleme auf Windows

Falls senity.ps1 nicht startet: Set-ExecutionPolicy -Scope CurrentUser RemoteSigned. Der .bat-Shim ruft pwsh mit -ExecutionPolicy Bypass auf, fuer cmd.exe ist das transparent.
